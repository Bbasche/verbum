The right abstraction
changes everything downstream.

Most frameworks model agent systems as function calls that happen to use language. You invoke tools. You query models. You stitch results together. It works — until you need to observe it, debug it, replay it, or hand control to a human.

Verbum starts from a different premise. Every participant is an Actor. Every interaction is a Message. Your terminal isn't a "tool" — it's in the conversation. Your MCP server isn't a plugin — it's a participant. Your human isn't a callback — they're first-class.

"What if every part of your system could talk to every other part — in a language you could actually read?"

The result: a system that is observable by construction, replayable by design, and composable without ceremony.

Seven actor types.
One universal interface.

⬡

ModelActor

Any LLM, wrapped uniformly. One line to swap providers.

  • Gemini (Google)
  • Ollama (local)
  • OpenRouter
▸

ProcessActor

A persistent shell session as a first-class participant. Any CLI, any subprocess, any REPL.

  • bash / zsh / fish
  • Python / Node REPL
  • Any binary CLI
  • Docker exec
⊛

MCPActor New

Speaks the Model Context Protocol natively. Auto-discovers tools from any MCP server on connect.

  • stdio servers
  • SSE servers
  • Auto capability registry
  • Claude Code compatible
◎

ToolActor

Deterministic functions and API wrappers. Tools that can also initiate conversations.

  • REST / GraphQL APIs
  • Pure functions
  • Webhooks
◉

HumanActor

A human is just another actor. Pause any flow, inject a response, resume — native to the model.

  • stdin / CLI
  • WebSocket
  • Slack / Webhook
  • Nostr (Phase 4)
◫

MemoryActor

Persistent context that participates in conversation. Ask it anything — it responds like an actor.

  • In-memory
  • SQLite
  • Qdrant (vector)
  • File (JSONL)
⟡

RemoteActor Phase 4

An actor on another machine. From the Router's perspective — indistinguishable from local.

  • Nostr (p2p)
  • WebSocket
  • NATS
  • HTTP
Show me the code

Same API. Every actor.

agent.ts 
import { Router, ModelActor, ProcessActor, MCPActor, MemoryActor } from 'verbum'

const router = new Router()

// Every participant — same interface, same message format
router.register(new ModelActor({ id: 'claude', provider: 'anthropic' }))
router.register(new ProcessActor({ id: 'shell', shell: '/bin/bash' }))
router.register(new MCPActor({
  id: 'filesystem', transport: 'stdio',
  command: 'npx', args: ['-y', '@modelcontextprotocol/server-filesystem', '.'],
}))
router.register(new MemoryActor({ id: 'memory', backend: 'sqlite' }))

// Send once. Claude routes to shell, MCP, and memory — automatically.
await router.send({
  from: 'user', to: 'claude', role: 'user',
  content: { type: 'text', text: 'Find the largest file, summarise it, save the summary' },
  conversationId: 'run-001',
})

// Every message recorded. Every edge in the graph. Fully replayable.
const graph = router.visualize()
graph.toMermaid()     // → diagram string
graph.replay()        // → AsyncIterable<Message>
graph.fork('msg-12') // → branch from any point
Why it matters

Four things you get for free
from the right abstraction.

01

Observable by construction

Every interaction — model to shell, agent to MCP server, human to model — is a structured, readable Message. Not a log line. Not a trace ID. The entire run is a conversation you can open and read.

02

Replayable. Forkable. Diffable.

Every run serialises as a conversation graph. Replay at any speed. Fork from any message — explore what would have happened if the model had responded differently. Debug with time travel.

03

Portable context

Move a conversation between providers mid-flight. Start with Claude, hand off to a local model, resume with GPT. The context belongs to no one — it's just a Message history.

04

Composable without ceremony

An agent that uses a shell, an MCP server, and a memory store is three registered actors and a routing rule. No pipelines to wire. No abstractions to fight.

05

Humans are first-class

Human-in-the-loop is native. Pause any flow mid-conversation, route a message to a HumanActor, resume when they respond. The graph spans both.

06

The full MCP ecosystem

MCPActor speaks the Model Context Protocol natively. Any MCP server — filesystem, browser, database — is an actor. If it works with Claude Code, it works with Verbum.

Phase 4 · Federated Mesh

Conversations that span
machines — without a server.

RemoteActor makes a participant on another machine indistinguishable from a local one. Route a message to gemini@npub1abc... and the framework delivers it — signed, optionally encrypted, over your transport of choice.

NostrTransport
Signed messages over Nostr relays. No server required. Humans with any Nostr client participate natively.
p2p · no server
WebSocketTransport
Direct node-to-node. Best for LAN or same-org deployments where you control both ends.
direct · low latency
NATSTransport
For teams running their own messaging infrastructure. High-throughput, durable, multi-subscriber.
self-hosted · durable
HTTPTransport
Webhook-style. Any HTTP endpoint becomes a reachable Verbum actor. Dead simple.
universal · stateless
federation.ts 
import { Router, ModelActor } from 'verbum'
import { RemoteActor, NostrTransport } from '@verbum/network'

const router = new Router()
router.register(new ModelActor({ id: 'claude', provider: 'anthropic' }))

// A Gemini instance on another machine — same API as local
router.register(new RemoteActor({
  id: 'remote-gemini',
  address: 'gemini@npub1xyz...',
  transport: new NostrTransport({ relays: ['wss://relay.damus.io'] }),
}))

// Route locally or across the mesh — identically
await router.send({ from: 'user', to: 'remote-gemini', content: { type: 'text', text: '...' } })
